# Адаптация Учебного Задания

## Исходная Идея

Учебное задание описывает небольшой сервис записи на звонок по мотивам Cal.com:

- посетитель видит свободные слоты;
- посетитель выбирает удобное время;
- посетитель оформляет запись;
- владелец календаря видит список предстоящих встреч;
- приложение собирается в Docker image и запускается в контейнере.

Главная учебная цель — пройти полный цикл разработки с ИИ-агентами: сначала зафиксировать поведение и API-контракт, затем отдельно реализовать frontend и backend, после этого покрыть ключевые сценарии тестами и подготовить Docker runtime.

## Наша Версия

Этот проект реализует базовый сценарий задания и добавляет production-like слой администрирования:

- публичный booking flow остается без авторизации;
- админка защищена входом по email и паролю;
- владелец календаря может смотреть и отменять встречи;
- владелец может менять доступность и публичный профиль;
- session token хранится в HttpOnly cookie, в базе хранится только hash token;
- пароль администратора хранится как Argon2 hash.

Авторизация не является обязательной частью исходного задания, но оставлена как расширение, потому что она делает owner/admin часть ближе к реальному сервису и дает дополнительные учебные темы: sessions, guards, secure cookies и protected API endpoints.

## Design First

Проект сохраняет Design First процесс:

1. `docs/api-contract.md` описывает endpoints, request/response shapes и ошибки.
2. Frontend в `apps/web` работает по этому контракту и не вычисляет booking rules самостоятельно.
3. Backend в `apps/api` реализует контракт, validation, auth и booking rules.
4. Vitest/Supertest tests проверяют API behavior.
5. Playwright e2e tests проверяют пользовательский сценарий в браузере.
6. Dockerfile собирает frontend и backend в один runtime container.

## Отличия От Минимального Задания

- Добавлена admin-auth.
- Добавлены настройки доступности и профиля.
- Поддержаны meeting categories на 15 и 30 минут.
- Production runtime остается одним контейнером, где NestJS отдает и API, и React build.

Эти отличия не меняют основной сценарий задания: гость выбирает свободный слот и записывается на звонок, а владелец календаря видит и управляет встречами.
