# Cal.com Research Notes

## Что Изучали

Cal.com используется как предметный и UX-референс для сервиса записи на звонок. Цель была понять базовый путь пользователя, а не копировать реализацию или интерфейс.

## Базовый User Flow

1. Владелец календаря создает тип встречи.
2. Владелец задает длительность и доступные интервалы.
3. Посетитель открывает публичную страницу бронирования.
4. Посетитель выбирает тип встречи.
5. Посетитель выбирает дату и свободный слот.
6. Посетитель вводит имя, email и комментарий.
7. Сервис показывает confirmation page.
8. Владелец видит встречу в списке бронирований.

## Что Перенесено В MVP

- Публичная страница записи.
- Выбор meeting category.
- Календарь с доступными слотами.
- Отображение свободных и занятых слотов.
- Форма гостя.
- Страница подтверждения.
- Admin список встреч.
- Отмена встречи владельцем календаря.

## Что Не Входит В MVP

- Внешние календарные интеграции.
- OAuth.
- Публичная регистрация.
- Команды и несколько владельцев календаря.
- Платежи.
- Email delivery.
- Автоматический перенос встречи гостем.

## Почему Нужен API-Контракт

Frontend и backend могут разрабатываться разными ИИ-агентами. Чтобы агентам не приходилось каждый раз читать всю чужую часть кода, поведение фиксируется в `docs/api-contract.md`.

При изменениях сначала обновляется контракт, затем реализация и тесты. Это снижает риск рассинхронизации UI и API.
